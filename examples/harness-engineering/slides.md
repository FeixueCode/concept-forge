# Harness 工程

你写的 haness 我按更常见的 harness 理解：Harness 工程就是给 AI 智能体做一套“马具/操作系统”，不是只教它一句 prompt，而是把工具、权限、上下文、检查和反馈都套好，让它能稳定干活。

# 它是什么

在 AI 语境里，harness 不是模型本身，而是包在模型外面的工作环境。它决定智能体能看见什么、能用哪些工具、什么时候停下来检查、犯错后如何恢复，以及人如何追踪它做过什么。

::right::

# 为什么重要

只靠更好的 prompt，往往只能改善一次回答；但真实任务需要多轮执行、调用工具、处理错误和交付结果。Harness 工程把这些外部条件做成系统，让智能体从“会聊天”变成“能被管理地做事”。

---

# 它怎么工作

<v-clicks>

- 先定义任务入口：用户目标、边界、完成标准和禁止事项。
- 再组织上下文：项目说明、历史决策、文件结构、当前状态和相关资料。
- 然后配置工具与权限：哪些命令能跑、哪些文件能改、哪些动作需要确认。
- 接着建立反馈回路：测试、校验、日志、审查、失败恢复和进度记录。

</v-clicks>

---

# 它不是什么

<v-clicks>

- 不是传统汽车或电子行业里的线束工程；那个更多是在设计电缆、连接器和布线路径。
- 不是单纯写一个更长的 Prompt；Prompt 只是 harness 里的一个小部件。
- 不是某个固定软件或平台；它是一套围绕智能体可靠运行的系统设计方法。
- 不是让 AI 完全自由发挥；恰好相反，它强调让 AI 在清楚边界里发挥。

</v-clicks>

---

# 相近概念怎么分

- **Prompt 工程**：Prompt 工程像写一张任务纸条；Harness 工程像搭一个完整工作台，纸条、工具、检查表和安全规则都在里面。
- **Context 工程**：Context 工程是 harness 的核心零件之一；Harness 还包括工具、权限、反馈、记忆和运行流程。
- **Agent Framework**：框架是材料和脚手架；Harness 工程是如何把材料组织成可运行、可检查、可维护的系统。

---

# 放进知识地图里

- **AI Agent**（上位概念）：Harness 工程服务的对象通常是需要连续执行任务的智能体。
- **Context Engineering**（下位概念）：管理上下文是 harness 中最关键的一层。
- **Tool Use**（下位概念）：智能体可靠性很大程度取决于工具接口是否清楚、有限、可验证。
- **Observability**（下位概念）：如果看不见智能体做了什么，就很难改进或追责。
- **Prompt Engineering**（容易混淆）：它们常被混为一谈，但层级不同。

# 一个具体例子

## 一个 Codex 例子

同样是让 AI 修改代码，只写一句“帮我改好”很容易失控；如果有 AGENTS.md、测试命令、文件边界、权限确认、进度记录和最终检查，这整套东西就是一个 harness。

# 三句话带走

<v-clicks>

- Harness 工程关注的是模型外面的系统，而不是模型本身。
- 它把上下文、工具、权限、反馈和记忆组织成可复用的运行环境。
- 它和 Prompt 工程的差别是：一个改一句话，一个改整套工作条件。

</v-clicks>
